**Objetivo:** Treinar a arquitetura Llama-like do tiny por 10× mais iterações (50k em vez de 5k) para calibrar hiperparâmetros e descobrir o ponto de saturação antes de investir no 20M.

- **Dataset:** `wikimedia/wikipedia` config `20231101.pt`, mesmos `tokens.npy` dos runs anteriores (593M tokens treino + 6M val).

## Análise

### Comparação com runs anteriores

| Run | Iters | Tokens treino | Best val loss | Melhora vs aleatório |
|---|---|---|---|---|
| tiny-v1-gpt2 | 5.000 | ~40M | 6.7514 | 3.0 nats |
| tiny-v2-llama | 5.000 | ~40M | 6.6408 | 3.1 nats |
| **tiny-long** | **50.000** | **~410M** | **6.3949** | **3.3 nats** |

### A descoberta central: saturação de capacidade

Com 10× mais iterações, val loss caiu só **0.25** (de 6.64 para 6.39). Em contraste, as primeiras 5000 iters já tinham tirado a loss de 9.68 (aleatório) para 6.64, uma queda de 3.0 nats. Visualmente na curva:

- **Steps 0-5.000:** val cai de 9.68 para 6.65 (rápido)
- **Steps 5.000-20.000:** val cai de 6.65 para 6.48 (lento)
- **Steps 20.000-50.000:** val vai de 6.48 para 6.39 (quase plano)

Isso é **capacity saturation**: o modelo de 3.2M params não-embedding literalmente não tem espaço para representar mais do corpus. Mais compute não resolve. Precisa de **mais parâmetros**.

### Validação empírica do Chinchilla

O ratio de tokens por parâmetro ficou em **172**, 8.6× acima do ótimo de 20. A degradação do ganho por step é exatamente o que a lei prevê: **para aproveitar um corpus deste tamanho, o modelo deveria ter ~25M parâmetros não-embedding**, o que é aproximadamente onde o Maracatu-20M se encaixa (17M total ≈ 10.6M não-embedding).

### Qualitativo: ganho real e perceptível

Apesar da loss ter ganho pouco em números absolutos, as amostras melhoraram de forma **visível**:

- Menos "palavra-salada" aleatória
- Narrativa flui entre orações ("por outro lado", "que já havia sido")
- Entidades reais aparecem com coerência contextual: **Rede Globo** em contexto de novela, Machado + **livro + editora + datas**, Pernambuco + **imperador + rei + paz** (contexto histórico, ainda que factualmente confuso)
- Estrutura de período composto aparece (`..., pela paz em 1798. em 1906, quando ...`)

A loss nominal é só marginalmente melhor, mas o que o modelo **faz** é bem melhor. Reminder de que loss não é o único sinal.

### Gap train/val

Manteve-se pequeno ao longo do treino (< 0.1 nats no plateau), sem overfitting detectável. Faz sentido: com 0.7 epoch e regularização fraca (dropout=0, weight_decay=0.1), o modelo não teve chance de memorizar.

### Implicações pro próximo passo (20M)

Este experimento valida três decisões:

1. **Arquitetura Llama-like está correta.** RoPE + SwiGLU + RMSNorm funcionam no nosso pipeline.
2. **Hiperparâmetros de LR/warmup/schedule estão bem calibrados.** A curva é saudável, sem instabilidade, sem divergência.
3. **Faltam parâmetros, não dados nem iteração.** O 20M (~10.6M não-embedding) é exatamente a escala certa para aproveitar o corpus de 550M tokens.

### Próximo passo

Partir para o **Maracatu-20M** sem mais experimentação a tiny. O tempo estimado de treino (~6h no M2 ou ~3h numa T4 do Kaggle) se justifica pelo ganho esperado em capacidade: 3.3× mais parâmetros não-embedding deve levar val loss a 4.5-5.0 (estimativa com base em scaling laws).
