# tiny-v1-gpt2 (2026-04-18)

**Git revision:** between `b8517ea` and `235c338` (before commit `766b936` that migrated the architecture)
**Config:** [`configs/maracatu_tiny.yaml`](../../configs/maracatu_tiny.yaml) (GPT-2-like version, already overwritten)
**Hardware:** MacBook Pro M2 Pro, 16 GB RAM, MPS backend
**Goal:** End-to-end validation of the pipeline (corpus → tokenizer → tokens.npy → training → sampling) with the original GPT-2-like architecture.

## Setup

- **Architecture:** GPT-2-like: LayerNorm, Multi-Head Attention with flash attention, GELU, absolute learned position embeddings, weight tying, `bias=False` on `nn.Linear` (LLaMA convention).
- **Params:** 5.06M total (≈ 3.2M non-embedding with pos emb subtracted).
- **Dataset:** `wikimedia/wikipedia` config `20231101.pt` (CC BY-SA 4.0). After filtering and dedup: 979,492 articles, 2.28 GB, ~550M BPE tokens.
- **Tokenizer:** SentencePiece BPE, vocab 16,000, `nmt_nfkc_cf` (lowercase), `split_digits=True`, `byte_fallback=True`.

## Analysis

**Verdict:** end-to-end pipeline validated. The model learned PT-BR syntax (articles, prepositions, agreement, frequent collocations like "densidade populacional", "a cidade possui") but didn't learn **facts**: 5000 iters over 7% of the corpus can't be expected to give more.

**Good signs:**
- Loss dropped monotonically (9.68 → 6.75)
- Small train/val gap (~0.08), no overfitting
- Generation produces syntactically valid PT-BR

**Expected signs:**
- Numbers are garbage (`85.10 e 366`), a consequence of `split_digits=True`
- Factual content invented (insufficient capacity + undertraining)
- Everything in lowercase: tokenizer normalization

**Limitation of this architecture:** 2020-era conventions (GPT-2-like): the state_dict doesn't match Hugging Face's `LlamaForCausalLM`, so it would need a conversion script for public publishing. Motivation for the migration documented in [tiny-v2-llama](2026-04-18-tiny-v2-llama.md).
